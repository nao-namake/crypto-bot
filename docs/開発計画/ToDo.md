## Phase 61: コードベース整理・戦略改修

### 背景と方針転換

Phase 61.1でレジーム閾値調整を試行したが**失敗**：
- PF: 1.58 → 1.13（-28%）
- 総損益: ¥86,639 → ¥21,781（-75%）
- trending発生率: 0% → 0.6%（目標5-15%に大幅未達）

**教訓**:
1. レジーム閾値変更だけでは市場特性は変えられない
2. 2025年下半期のBTC/JPYは本質的にtrendingが少ない
3. 閾値変更は取引パターン全体に影響し、リスクが高い

**新方針**: レジーム改修は断念し、**既存環境（tight_range 88%）での戦略最適化**に注力

### Phase 61.5: 低信頼度エントリー対策 📋予定

**背景**:
- 2026年1月25日、3連続SL発動（-460円, -484円, -346円）
- 下落相場でロングエントリー、信頼度0.194（19.4%）という異常値で取引実行
- MLはHOLD（0.577）を予測したが、ML信頼度が`high_confidence_threshold`（0.70）未満のためペナルティ未適用

**調査結果（7日間200サンプル）**:

| 信頼度範囲 | 件数 | 割合 |
|-----------|------|------|
| 0.40以上（高） | 192件 | 96% |
| 0.25-0.40（中） | 0件 | 0% |
| **0.25未満（低）** | **8件** | **4%** |

- 最小: 0.184、最大: 0.590、平均: 0.424
- 通常のエントリーは0.40-0.45に集中
- **0.25未満は異常値であり、対策しても96%のエントリーには影響しない**

**問題箇所**: `src/core/services/trading_cycle_manager.py`
- ML統合後の`adjusted_confidence`に最低閾値チェックがない
- ML信頼度 < 0.70の場合、HOLD変換判定がスキップされる

**解決策**:
1. `thresholds.yaml`に`min_strategy_confidence: 0.25`を追加
2. `trading_cycle_manager.py`で最終信頼度チェック追加
   - `adjusted_confidence < min_strategy_confidence` → 強制HOLD

**実施内容**:
1. thresholds.yamlに設定追加
2. trading_cycle_manager.pyの`_apply_ml_integration()`修正
3. 単体テスト追加
4. バックテストで影響確認（エントリー数4%減、PF維持を確認）

**成功基準**:
- 0.25未満の異常エントリーが0件になる
- PF ≥ 1.55維持
- エントリー数の大幅減少なし（4%程度の減少は許容）

## Phase 62: 逆行対策・利益確保改善（予定）

### 背景（Phase 61.4 MFE/MAE分析結果）

Phase 61.4のMFE/MAE分析で判明した問題:
- 負けトレード157件中156件（99%）が一度は利益だった
- 平均MFE: ¥753（利益方向）→ 実際のPnL: ¥-948（損失）
- **¥266,919の利益を逃している**

**真の問題**: TPに到達せず、利益から逆行して損失になるパターン

| 62.3 | 部分決済機能 | 低 | 50%到達で半分決済 |

### Phase 62.1: トレーリングストップ実装 📋予定

**目的**: 一度利益が出たポジションの逆行時に自動決済

**設計案**:
```yaml
trailing_stop:
  enabled: true
  activation_threshold: 0.002  # 0.2%利益でトレーリング開始
  trail_distance: 0.001        # 0.1%の距離でトレーリング
```

**実装箇所**:
- `src/trading/execution/stop_manager.py` - トレーリングロジック追加
- `config/core/thresholds.yaml` - 設定追加

**成功基準**:
- 負けトレードのMFE捕捉率向上（現在0% → 目標30%以上）
- PF維持（≥ 1.55）

---

### Phase 62.2: 時間ベース決済 📋予定

**目的**: 長時間保有による逆行リスクを軽減

**設計案**:
- 保有時間 > N時間 かつ 含み益 > 0 → 決済
- N = 4〜8時間を検討

**判断材料**:
- 現在の平均保有時間の分析
- 保有時間と勝率の相関分析

---

### Phase 62.3: 部分決済機能 📋予定

**目的**: 利益の一部を確定しつつ、残りでさらなる利益を狙う

**設計案**:
- 含み益が一定額（例: ¥500）に到達 → 50%決済
- 残り50%は元のTP/SLで継続

**注意点**:
- bitbank APIで部分決済がサポートされるか確認必要
- 手数料の影響を考慮
